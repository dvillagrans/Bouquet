\section{Metodología de Desarrollo}

\subsection{Enfoque metodológico}

El desarrollo del presente proyecto se plantea mediante un enfoque híbrido que combina
Kanban iterativo para la gestión visual e incremental del trabajo y una metodología de
ingeniería de datos basada en arquitectura Medallion para el diseño, procesamiento
y validación del componente analítico.

La elección de este enfoque responde a la naturaleza dual del sistema. Por un lado, se
requiere construir una plataforma web progresiva con múltiples roles, mesas virtuales,
órdenes, cocina y liquidación interna. Por otro lado, el núcleo técnico del Trabajo
Terminal se centra en el diseño de la base de datos transaccional, la trazabilidad de
eventos operativos, el procesamiento batch con Apache Spark y la generación de
indicadores para inteligencia de negocio.

Kanban resulta adecuado para un equipo académico reducido, ya que permite visualizar el
flujo de trabajo, limitar tareas en progreso, priorizar entregables y dar seguimiento
continuo sin requerir roles formales de Scrum. La metodología Medallion, por su parte,
estructura el flujo analítico en capas Bronze, Silver y Gold, garantizando trazabilidad
entre los datos transaccionales originales y las métricas presentadas en el dashboard
administrativo.

\subsection{Kanban iterativo para la gestión del proyecto}

El proyecto utilizará Kanban como enfoque ágil ligero para organizar, priorizar y dar
seguimiento a las actividades del Trabajo Terminal. Kanban se basa en la visualización
del flujo de trabajo, la limitación del trabajo en progreso y la mejora continua del
proceso \cite{anderson_kanban_2010}. Esta metodología resulta adecuada para equipos reducidos, ya que no requiere
la asignación estricta de roles como Product Owner, Scrum Master o Development Team.

Para el presente proyecto, el tablero Kanban se organizará en las siguientes columnas:
\textit{Backlog}, \textit{Por hacer}, \textit{En progreso}, \textit{En revisión},
\textit{Validado} y \textit{Terminado}. Cada tarjeta representará una actividad funcional,
técnica, documental o analítica, asociada a un entregable verificable.

Aunque no se aplicarán ceremonias Scrum formales, el equipo realizará revisiones
periódicas quincenales para evaluar avances, validar entregables, ajustar prioridades
y registrar observaciones de la directora del Trabajo Terminal. De esta forma, el
proyecto conserva ciclos de retroalimentación frecuentes sin generar sobrecarga
metodológica innecesaria para un equipo de dos integrantes.

La gestión mediante Kanban se justifica por tres razones principales:

\begin{itemize}[leftmargin=1.5em]
  \item Equipo reducido. Al estar conformado por dos integrantes, el proyecto
  requiere una metodología flexible que permita coordinación directa sin roles
  ceremoniales rígidos.

  \item Naturaleza mixta del trabajo. El proyecto incluye actividades de
  investigación, documentación, diseño, desarrollo de software, modelado de datos,
  procesamiento analítico y pruebas, por lo que resulta conveniente gestionar tareas
  de distinto tipo dentro de un mismo flujo visual.

  \item Entregables verificables. Cada tarjeta del tablero deberá producir
evidencia concreta, como diagramas, migraciones, interfaces, endpoints, datasets,
  jobs Spark, tablas analíticas, pruebas o documentación.
\end{itemize}

Aunque no se adopta XP como metodología principal, se incorporan prácticas compatibles
como revisión cruzada, integración continua, pruebas funcionales y refactorización
controlada cuando sea necesario.

\subsection{Metodología de ingeniería de datos}

El componente analítico del sistema se desarrollará siguiendo una metodología de ingeniería
de datos basada en arquitectura Medallion. Este enfoque organiza el procesamiento en capas
progresivas de refinamiento, permitiendo transformar registros transaccionales en datos
analíticos confiables.

\begin{enumerate}[leftmargin=1.5em]
  \item Identificación de eventos fuente.
  Se definirán las entidades y eventos operativos que alimentarán el análisis, tales como
  apertura de mesas virtuales, registro de órdenes, productos consumidos, cambios de estado
  en cocina, comensales asociados y registros de liquidación interna.

  \item Captura transaccional.
  Los eventos se almacenarán en PostgreSQL mediante un modelo relacional diseñado para
  preservar integridad referencial, consistencia y trazabilidad histórica. Se considerarán
  snapshots de productos y precios para evitar que cambios futuros en el catálogo alteren
  órdenes pasadas.

  \item Extracción hacia capa Bronze.
  Se generarán datasets crudos a partir de la base transaccional, conservando los registros
  originales, sus identificadores y marcas temporales.

  \item Transformación hacia capa Silver.
  Mediante Apache Spark se realizará limpieza, normalización y enriquecimiento de datos,
  incluyendo validación de fechas, montos, relaciones entre entidades y estados operativos.

  \item Agregación hacia capa Gold.
  Se construirán tablas analíticas orientadas a indicadores de negocio, tales como ventas
  por sucursal, productos más consumidos, actividad por franja horaria, tiempos de preparación
  y comportamiento de liquidación interna.

  \item Visualización y validación.
  Las métricas generadas se visualizarán en el dashboard administrativo y se validarán
  comparando los resultados agregados contra los datos transaccionales de origen.
\end{enumerate}

\subsection{Planificación por iteraciones}

El desarrollo se organizará en 12 iteraciones de aproximadamente dos semanas,
considerando ajustes puntuales por revisiones académicas, entregas parciales y
validación con la directora del Trabajo Terminal. Cada iteración agrupa actividades
funcionales, técnicas o analíticas que serán gestionadas mediante el tablero Kanban.

\begin{enumerate}[leftmargin=*, itemsep=1pt, topsep=2pt]
  \item Iteración 1: Análisis de reglas de negocio, eventos transaccionales y métricas analíticas.
  \item Iteración 2: Diseño de base de datos transaccional, MER y modelo lógico.
  \item Iteración 3: Implementación de schema Prisma, migraciones y datos semilla.
  \item Iteración 4: Diseño UX, wireframes y flujos por rol.
  \item Iteración 5: Autenticación, roles, RLS y estructura organizacional.
  \item Iteración 6: Catálogo, mesas físicas, mesa virtual y acceso QR.
  \item Iteración 7: Órdenes, canasta, snapshots históricos y tablero de cocina.
  \item Iteración 8: Liquidación interna, aportaciones y control de concurrencia.
  \item Iteración 9: Extracción batch y capa Bronze del pipeline analítico.
  \item Iteración 10: Transformaciones Spark, capa Silver y reglas de calidad de datos.
  \item Iteración 11: Capa Gold, KPIs y dashboard administrativo.
  \item Iteración 12: Pruebas integrales, validación de métricas, seguridad y cierre documental.
\end{enumerate}

\subsection{Criterios de terminado}

Para considerar terminado un incremento de trabajo, deberá cumplir con los siguientes
criterios:

\begin{itemize}[leftmargin=1.5em]
  \item Código integrado en la rama correspondiente y sin errores críticos de compilación.
  \item Migraciones de base de datos aplicadas y documentadas cuando corresponda.
  \item Pruebas funcionales básicas ejecutadas sobre el módulo desarrollado.
  \item Validación de permisos y restricciones de acceso si el módulo involucra usuarios o roles.
  \item Evidencia visual o técnica del incremento: capturas, endpoint funcional, tabla generada,
  dataset procesado o dashboard actualizado.
  \item Actualización de documentación técnica y bitácora de avance.
\end{itemize}

En los incrementos analíticos, además se deberá comprobar la correspondencia entre los
registros transaccionales de origen y las capas Bronze, Silver o Gold generadas.

\subsection{Validación de entregables}

La validación del proyecto se realizará mediante evidencia funcional, técnica y analítica.
Los módulos operativos se validarán mediante pruebas de flujo por rol, considerando
comensal, mesero, cocina y administración. El modelo de datos se validará mediante
integridad referencial, migraciones exitosas y registros de prueba.

El pipeline analítico se validará mediante la comparación entre datos transaccionales de
origen y datasets Bronze, Silver y Gold generados por Apache Spark. Finalmente, el dashboard
se validará comprobando que sus métricas correspondan con las tablas analíticas construidas
y con los datos registrados en la operación.


\subsection{Cronograma general}

El proyecto abarca del 1 de enero de 2026 al 30 de noviembre de 2026 y se organiza
en frentes de trabajo iterativos, no en fases aisladas por integrante. Aunque cada
frente cuenta con un liderazgo principal según la experiencia de los autores, ambos
participan en la revisión, validación y documentación de los entregables.

Durante los primeros meses se concentra el análisis del dominio, la revisión de
literatura, la definición de requisitos y el diseño inicial del modelo de datos.
Posteriormente, el trabajo avanza hacia la implementación de la plataforma, el pipeline
analítico, la validación de métricas y las pruebas integrales. La documentación se
mantiene como una actividad transversal durante todo el proyecto.
